# XP（エクストリームプログラミング）研修 - Java練習問題

## 概要

このプロジェクトは、社内の若手エンジニア向けのXP（エクストリームプログラミング）研修用の練習問題です。
Java 17とMavenを使用して、XPの5つの価値、原則、プラクティスを実際のコーディングを通して学習します。

## XPの基本理念

### XPの5つの価値（Values）

1. **コミュニケーション（Communication）**
   - チームメンバー間の積極的な対話
   - 問題の早期発見と解決
   - 知識の共有

2. **シンプルさ（Simplicity）**
   - 必要最小限の機能実装
   - 複雑さの回避
   - 理解しやすいコード

3. **フィードバック（Feedback）**
   - 早期で頻繁なフィードバック
   - テストによる即座の結果確認
   - 顧客からの継続的な意見

4. **勇気（Courage）**
   - 必要な変更を恐れない
   - 問題に正面から取り組む
   - リファクタリングの実行

5. **尊敬（Respect）**
   - チームメンバーへの敬意
   - 多様な意見の受け入れ
   - 協力的な関係構築

### XPの原則（Principles）

**基本原則**:
- **小さなステップ（Baby Steps）**: 変更を小さく分割して進める
- **変更の受け入れ（Embracing Change）**: 変更を歓迎し、適応する
- **品質への責任（Quality Work）**: 高い品質基準を維持する

**実践原則**:
- **漸進的変更（Incremental Changes）**: 段階的な改善
- **試験による証明（Play to Win）**: テストで品質を保証
- **適応（Adaptation）**: 状況に応じた調整
- **多様性（Diversity）**: 異なる視点の活用
- **責任の受け入れ（Accepted Responsibility）**: 自分の作業に責任を持つ

## 練習問題

### 1. FizzBuzz - TDD（テスト駆動開発）練習

**学習目標**: Test-First Development、Simple Design、Refactoring

**課題**:
- `src/main/java/com/company/xp/FizzBuzz.java`を実装
- TDDサイクル（Red-Green-Refactor）を実践
- 最初にテストを書き、テストが通る最小限のコードを実装

**XPの価値を体験**:
- **シンプルさ**: 必要最小限の機能のみ実装
- **フィードバック**: テストによる即座のフィードバック
- **勇気**: リファクタリングを恐れずに実行

**XPプラクティス**:
- テストファースト
- シンプルな設計
- 継続的なリファクタリング

### 2. Calculator - ペアプログラミング練習

**学習目標**: Pair Programming、Collective Code Ownership

**課題**:
- `src/main/java/com/company/xp/Calculator.java`を実装
- 2人1組でドライバー/ナビゲーターを交代しながら開発
- 5-10分間隔でロールを交代

**XPの価値を体験**:
- **コミュニケーション**: 積極的な対話とアイデアの共有
- **尊敬**: パートナーの意見を尊重し、建設的な議論
- **勇気**: 相手に質問し、自分の考えを表現する勇気

**XPの原則を実践**:
- **多様性**: 異なる視点とアプローチの活用
- **責任の受け入れ**: ペアとしての品質責任

**XPプラクティス**:
- ペアプログラミング
- コードの共同所有
- コミュニケーション重視

### 3. StringCalculator - リファクタリング練習

**学習目標**: Refactoring、Small Releases

**課題**:
- `src/main/java/com/company/xp/StringCalculator.java`を実装
- 機能を段階的に追加（Small Releases）
- 各段階でリファクタリングを実施

**XPの価値を体験**:
- **シンプルさ**: 各ステップでシンプルな設計を維持
- **フィードバック**: 各機能追加後にテストで確認
- **勇気**: 既存コードの改善を恐れずに実行

**XPの原則を実践**:
- **小さなステップ**: 機能を小さく分割して実装
- **漸進的変更**: 段階的な機能追加
- **変更の受け入れ**: 要件変更に柔軟に対応

**実装ステップ**:
1. 空文字列の処理
2. 単一数字の処理
3. カンマ区切り数字の処理
4. 改行文字対応
5. カスタム区切り文字対応
6. 負の数の検証
7. 1000以上の数字を無視

**XPプラクティス**:
- 継続的リファクタリング
- 小さなリリース
- シンプルな設計

### 4. BowlingGame - 継続的統合練習

**課題**:
- `src/main/java/com/company/xp/BowlingGame.java`を実装

**XPの価値を体験**:
- **フィードバック**: テストによる継続的な品質確認
- **シンプルさ**: 複雑な要件をシンプルに分解
- **勇気**: 複雑なロジックに立ち向かう

**XPの原則を実践**:
- **品質への責任**: 高いテストカバレッジの維持
- **小さなステップ**: 機能を小さく分割して実装
- **適応**: テスト結果に基づく継続的な改善

**XPプラクティス**:
- 継続的統合
- 包括的なテスト
- 小さなリリース

## XPプラクティスガイド

### TDD（テスト駆動開発）の流れ

1. **Red**: 失敗するテストを書く
2. **Green**: テストが通る最小限のコードを書く
3. **Refactor**: コードを改善する（テストは通ったまま）

### ペアプログラミングのコツ

- **ドライバー**: コードを書く人
- **ナビゲーター**: 指示・レビューする人
- 定期的にロールを交代（5-10分）
- 積極的にコミュニケーションを取る
- 相手を尊重し、建設的な議論を心がける

### リファクタリングの原則

- 動作を変えずにコードの構造を改善
- 小さなステップで実施
- 各ステップでテストが通ることを確認
- コードの可読性と保守性を向上

### 継続的統合のベストプラクティス

- 頻繁にコミット
- すべてのテストが通ることを確認
- ビルドエラーは即座に修正
- 統合の問題を早期発見

## 学習の進め方

1. **各問題を順番に取り組む**
   - FizzBuzz → Calculator → StringCalculator → BowlingGame

2. **XPの価値・原則を意識する**
   - 単にコードを書くのではなく、5つの価値と原則の実践を重視
   - 各練習問題で重点的に体験する価値を意識する

3. **ペアやグループで取り組む**
   - 一人で取り組まず、チームで協力して学習
   - コミュニケーションと尊敬の価値を実践

4. **定期的な振り返りを行う**
   - 各問題完了後に、`XPReflection.java`を参考に価値・原則を振り返る
   - `XPAssessmentTest.java`のディスカッション課題に取り組む
   - 学んだことや改善点を議論

5. **継続的な改善**
   - 完成後も定期的にコードを見直し、改善の余地を探す
   - 小さなステップでの継続的な改善を実践

## トラブルシューティング

### よくある問題

1. **テストが実行されない**
   - JUnit 5の依存関係が正しく設定されているか確認
   - `@Test`アノテーションが正しくインポートされているか確認

2. **Java 17が認識されない**
   - `JAVA_HOME`環境変数が正しく設定されているか確認
   - IDE の Java バージョン設定を確認

3. **Mavenでビルドエラー**
   - `mvn clean`を実行してからビルドを試す
   - 依存関係の競合がないか確認
